# Headway 自動化內容擷取工具：開發總結

這份文檔總結了從零開始開發「Headway 書籍摘要抓取腳本」的歷程，記錄了期間遇到的各種技術挑戰、核心問題分析及其解決方案。

---

## 🏗️ 專案概況
本專案開發了一個基於 **Python + Playwright** 的自動化工具。該工具能讓使用者在瀏覽 Headway 網頁版時，透過點擊一個動態注入的「啟動」按鈕，實時抓取當下書籍的封面圖片、多頁摘要內文、作者資訊，並將其自動整理、過濾雜訊後存入對應的書名資料夾中。

---

## 🔍 腳本問題分析與修復 (關鍵轉折點)

在開發早期，腳本曾面臨完全無法下載的問題。以下是核心問題診斷與修復對照：

### ❌ 原始問題診斷
1. **Blob URL 下載無法被 Playwright 攔截**：腳本在瀏覽器端採用的 `URL.createObjectURL(blob)` 產生的是 `blob:` 協議。Playwright 的 `page.on("download", ...)` 僅能攔截真實的 HTTP 下載請求，無法識別瀏覽器內部的 Blob 行為。
2. **按鈕注入時機過早**：在 SPA (單頁應用程式) 下，React/Vue 可能在按鈕注入後重新渲染，導致按鈕消失。
3. **登入跳轉不明確**：僅使用 `time.sleep` 等待登入，忽視了網路波動與異步跳轉的變數。
4. **Download Handler 失效**：由於問題 1 的存在，導致處理下載的函數永不觸發。

### ✅ 修復方案與原則
改用 **「Python 端直接擷取頁面內容」** 取代「瀏覽器端 Blob 下載」：

| # | 原始問題 | 修復方式 |
| :--- | :--- | :--- |
| 1 | blob: URL 無法被攔截 | 改由 Python 端透過 `page.evaluate()` 傳回字串，直接在本地寫檔 |
| 2 | 按鈕注入後被 SPA 覆蓋 | 改用頻繁偵測的注入機制與 `add_init_script` (最終回推為穩定的監聽器) |
| 3 | 登入等待時間不固定 | 使用 `page.wait_for_load_state("networkidle")` 確保渲染完成 |
| 4 | 下載事件不觸發 | 徹底移除下載攔截器，改為直接由 Python 負責寫檔 |
| 5 | 前後端通知不靈敏 | 使用全域旗標 `window.__headway_ready__` 搭配 Python 輪詢偵測點擊 |

**💡 核心原則**：瀏覽器端的 Blob 下載是「瀏覽器內部行為」，正確做法是讓 `page.evaluate()` 將資料傳回 Python，再由 Python 處理檔案系統操作。

## 🚀 最終功能清單
- **動態按鈕注入**：按鈕隨頁面自動重啟，下移 110px 避開導覽列，不擋使用者操作。
- **動態書名與封面偵測**：不再依賴寫死的 URL，點擊瞬間捕捉當下頁面的書名與網址。
- **深度多頁抓取**：遍歷 `summary?page=N` 直到「結論 (Conclusion)」或內容重複為止。
- **資料深度清理**：自動過濾導航 UI (Discover, Library...)、重複書名與音軌時間戳 (00:00)。
- **組織化儲存**：每本書建立獨立資料夾，同時下載 `cover.jpg` 封面與 `Summary_XXX.md` 筆記。
- **連續作業模式**：支援無限循環抓取，抓完一本可直接導覽至下一本繼續點擊下載。
